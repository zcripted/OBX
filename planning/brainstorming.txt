# SF-Core Holograms — Master Implementation Plan
## Single-Session, Phase-Continuous Build Plan for Claude Code

This document is the complete blueprint for adding a hologram subsystem to SF-Core.
It is written so a single Claude Code session can execute every phase back-to-back
without operator intervention, while preserving every existing SF-Core system and
matching every convention already in the codebase.

**Zero external plugin dependencies.** No ProtocolLib. No PacketEvents. No
DecentHolograms shim. Display entities are used natively (Paper API, reflection-gated)
and a thin custom Netty-channel layer handles inbound interaction packets when
finer-than-raycast accuracy is required.

---

## 0. Executive Summary

* **Module name:** `sfcore-holograms` (internal to SF-Core, not a separate plugin).
* **Package root:** `dev.sergeantfuzzy.sfcore.hologram`
* **Service entry point:** `HologramService` (load/reload/save lifecycle, matching
  `HubService`, `EnchantService`, `ChatService`, `TablistService`, `ScoreboardService`).
* **Master flag:** `systems/holograms.yml → enabled: false` (dormant by default,
  identical pattern to `systems/hub.yml`).
* **Backends (picked at runtime via `PlatformInfo`):**
    1. **Display-entity backend** — Paper/Spigot ≥ 1.19.4, uses real
       `TextDisplay` / `BlockDisplay` / `ItemDisplay` entities driven by Bukkit
       schedulers, visibility managed through `Player#hideEntity` / `Player#showEntity`.
    2. **Armor-stand backend** — Paper/Spigot 1.12 → 1.19.3, invisible marker
       armor stands with custom names. Same public API, reduced feature set
       (no per-stack icons in-line, no native rotation animation — emulated).
    3. **Custom packet layer (optional, advanced)** — a Netty `ChannelDuplexHandler`
       injected per-player via reflection of the underlying Mojang `Connection`
       channel. Used **only** for inbound `ServerboundInteractPacket` decoding so
       holograms can react to genuine click packets when raycast is insufficient.
       Outbound packets are **not** synthesized by hand — we let the server render
       the real entities. Degrades silently to raycast-only on any platform where
       reflection probes fail.
* **Continuous execution:** Phases 0 through 7 run in one session. Each phase
  ends with a build, a `docs/changes/` entry, language-file updates, and a
  green light to start the next phase. Phases never block on user approval.

## 1. Non-Negotiable Constraints (read FIRST)

These constraints override every later instruction. If a later instruction
appears to contradict one of these, the constraint wins.

### 1.1 SF-Core integration invariants
1. **Single plugin, one `JavaPlugin`.** All wiring happens in
   `dev.sergeantfuzzy.sfcore.Main`. There is no `SfHolograms.java`. No second
   plugin descriptor. No second `plugin.yml` block.
2. **Service-shaped, dormant by default.** `HologramService` mirrors
   `HubService` exactly: constructed in `Main#onEnable`, `load()` called once,
   `save()` called from `onDisable`, `reload()` exposed from `Main#reloadPlugin`
   and surfaced in the per-component reload timing map.
3. **Existing infrastructure must be reused, not duplicated.** Specifically:
    * Scheduling → `SchedulerAdapter` (already Folia-aware, already exposes
      `runAtEntity` / `runAtEntityLater` / `runAtLocation` — use them).
    * Platform / version detection → `PlatformInfo` (already exposes
      `isAtLeast`, `isPaper`, `hasAdventureApi`).
    * Resource pack → `AutoResourcePackManager` (image-line glyph font, if ever
      shipped, lives in the bundled pack — not a side-load).
    * Storage paths → mirror `WarpService` / `MotdService`: a service-owned YAML
      file under the plugin data folder, atomic save with `LocationSerializer`.
    * Language → `LanguageManager`. **Every** new user-visible string lands in
      both `language_en.yml` and `sprache_de.yml` in the same change.
    * Console logging → `dev.sergeantfuzzy.sfcore.util.message.ConsoleLog`.
    * Documentation → every phase emits a `docs/changes/YYYY-MM-DD---*.md` file
      following the template in `CLAUDE.md`, and edits
      `docs/information/COMMANDS+PERMISSIONS.md` when commands/perms change.
4. **No breaking changes to existing source.** No renames in existing packages,
   no signature changes on existing services, no removals. New code is purely
   additive. The one explicit `Main.java` touch is six new lines in `onEnable`
   plus matching lines in `onDisable` and `reloadPlugin`, modelled on
   `hubService` and `enchantService`.
5. **Compile against Spigot 1.12.2 API; run on 1.12 → 1.21+.** All display-entity
   classes are referenced through reflection or behind a runtime guard. No
   `import org.bukkit.entity.TextDisplay` at file scope unless the import is
   isolated in a class that is only loaded when `PlatformInfo#isAtLeast(1, 19, 4)`
   returns true.

### 1.2 No external plugin/library dependencies
1. **No new Maven dependencies** beyond what `pom.xml` already declares.
2. **No ProtocolLib, no PacketEvents, no shaded packet helpers.** All packet-
   adjacent work uses (a) Bukkit/Paper APIs for outbound entity state and
   (b) reflection-only Netty channel injection for inbound interaction packets.
3. **No NMS imports.** Everything that crosses into Mojang-mapped or
   `net.minecraft.server.*` territory does so through `Class.forName` /
   `Method.invoke` / `Field.get`, with the lookup cached on the
   declaring class. The platform layer already follows this pattern; the
   packet layer must too.
4. **Reflection failures are non-fatal.** Any probe that fails downgrades the
   subsystem to its next backend or feature tier. The service must *never*
   throw out of `load()` because of a missing class.

### 1.3 CLAUDE.md rules that apply to every phase
* Maven build via `./maven/bin/mvn -DskipTests package` (the in-project
  distribution; no global `mvn`).
* GUI item hover tooltips must be detailed, cleanly formatted, with dividers
  and spacing where appropriate.
* `docs/changes/` entries use the America/Detroit timestamp template.
* `docs/information/COMMANDS+PERMISSIONS.md` Player/Admin tables get updated
  in any phase that adds or changes a command or permission node.

### 2.1 Backend contract

```java
public interface HologramBackend {
    /** Spawn the entities backing this hologram in the world for everyone currently in range. */
    void spawn(Hologram hologram, Collection<Player> initialViewers);

    /** Re-evaluate viewers; show/hide as needed. Called every tick by ViewerTracker. */
    void updateVisibility(Hologram hologram, Player viewer, boolean shouldSee);

    /** Push any pending text / icon / scale / billboard changes to the live entities. */
    void applyMutations(Hologram hologram);

    /** Despawn for everyone, free entity ids. */
    void destroy(Hologram hologram);

    /** Optional: report whether the backend can do per-player line text. */
    boolean supportsPerViewerText();

    /** Diagnostic label printed in /sfholo info and the startup banner. */
    String describe();
}
```

Backends are stateless across holograms. Per-hologram state (entity refs, last
viewer set, dirty flag) lives in the `Hologram` model. Backends only **read**
from the model and call Bukkit APIs.

### 2.2 Why two backends, not one
* Bukkit 1.12.2 has no `TextDisplay` class — armor stands are the only option.
* Display entities are vastly cheaper (no AI, no physics, no nameplate label
  tied to a baby armor-stand head), so we prefer them whenever the server
  version allows.
* Both backends sit behind the same `HologramRenderer`, so commands /
  persistence / animations / interactions don't change.

### 2.3 Why a custom packet layer at all
Most operations don't need it:
* **Rendering** — handled by spawning real entities; the server packets them.
* **Visibility** — `Player#hideEntity` / `showEntity` (Paper API; reflected on
  Spigot ≥ 1.16, raycast fallback below that).
* **Per-player text** — when truly needed, spawn one display entity per viewer
  and hide it from others. Cheap because display entities have no AI.

The packet layer is used **only** for one thing: catching client click packets
on holograms. The display entity vanilla "click" event doesn't fire on
`Interaction` entities older than the version we run on, and raycast is
~0.25s laggy on the look-at side. So:

* The injector hooks each player's Netty channel on join.
* The `ChannelDuplexHandler` inspects inbound packets only.
* If the inbound packet is a `ServerboundInteractPacket` whose target id is a
  hologram-owned entity id, the handler queues an `HologramClickEvent` onto
  the main thread via `SchedulerAdapter#runNow`.
* Outbound traffic passes through unmodified.

This entire layer is wrapped in a `PacketAvailability` flag. If reflection
probes fail at startup, the layer reports `unavailable` once, the renderer
falls back to look-direction raycast, and the system keeps working.

### 3.1 Exact `Main.java` patch shape (preview for the agent)

In `Main#onEnable`, immediately after the enchant block:

```java
// Holograms module. Dormant when systems/holograms.yml master enabled=false.
hologramService = new dev.sergeantfuzzy.sfcore.hologram.service.HologramService(this);
hologramService.load();
```

In `Main#onDisable`, before `printBanner(false)`:

```java
if (hologramService != null) {
    hologramService.save();
}
```

In `Main#reloadPlugin`, before the closing `return times;`:

```java
if (hologramService != null) {
    s = System.nanoTime(); hologramService.reload(); times.put("holograms.yml", System.nanoTime() - s);
}
```

In `Main#registerCommands` and `Main#registerListeners`, one binding each — the
agent adds them when Phase 2 (commands) lands. Every other system in `Main` is
optional / nullable in disable; holograms follow that same nullability pattern.

### Phase 1 — Backend implementations + visibility

**Goal.** Both backends render a text line + an icon line + a block backing
in-world, with range-based visibility and double-sided culling. No persistence,
no commands — a hardcoded debug hologram from a temporary `/sfholo debug` stub.

**Files to create.**
* `hologram/backend/DisplayEntityBackend.java` — gated on
  `PlatformInfo#isAtLeast(1, 19, 4)`. Uses **reflection** to resolve
  `org.bukkit.entity.TextDisplay`, `BlockDisplay`, `ItemDisplay`,
  `org.bukkit.entity.Display.Billboard` enum, `setBillboard`, `setText`,
  `setBlock`, `setItemStack`, `setBackgroundColor`, `setTextOpacity`,
  `setSeeThrough`, `setShadowed`, `setAlignment`, `setLineWidth`, transform
  setters. Caches the `Method`/`Field` handles statically.
* `hologram/backend/ArmorStandBackend.java` — `world.spawn(loc, ArmorStand.class)`
  with `setInvisible(true)`, `setMarker(true)`, `setSmall(true)`,
  `setCustomNameVisible(true)`, `setCustomName(...)`. For icon lines the stand
  wears the item on its head with a stacked offset. For block lines the stand
  spawns above a falling-block helper (animation-stable, then converted to a
  display-stand later).
* `hologram/model/Hologram.java`, `HologramLine.java` (with inner classes
  `TextLine`, `IconLine`, `BlockLine`), `HologramBoard.java`,
  `HologramSettings.java`.
* `hologram/render/HologramRenderer.java`, `ViewerTracker.java`, `TickLoop.java`.
* `hologram/listener/HologramJoinListener.java`,
  `HologramResourcePackListener.java`.
* `hologram/command/HologramCommand.java` — minimal `debug` subcommand only.

**Files to edit.**
* `Main.java` — register `HologramCommand` (bound to `sfholo`),
  register `HologramJoinListener` and `HologramResourcePackListener`.
* `plugin.yml` — wire the `sfholo` command and its base permission node
  `sfcore.holo.use`.
* `language_en.yml` / `sprache_de.yml` — `hologram.debug.spawned`,
  `hologram.debug.removed`, `hologram.error.backend_unavailable`.

**Backend selection rules.**
* If `PlatformInfo#isAtLeast(1, 19, 4)` AND `TextDisplay` class resolves →
  `DisplayEntityBackend`.
* Otherwise → `ArmorStandBackend`.
* The decision is logged once at `HologramService#load`. The chosen backend's
  `describe()` is included in `/sf` plugin-info output.

**Visibility rules.**
* `ViewerTracker` ticks every N ticks (config: `view-update-ticks`, default 5).
* For each `Player` in the same world as the hologram, compute distance²;
  in-range = `shouldSee = true`.
* If `doubleSided == false`, additionally compute the dot product between
  the player's look direction and the hologram normal; flip to `false` when
  the viewer is behind the hologram plane.
* `applyMutations` only fires if the hologram's `dirty` flag is set.
* All tick work uses `SchedulerAdapter#runRepeating` and, on Folia, location-
  bound work uses `runAtLocation` so multi-world / multi-region servers stay
  thread-safe.

**Verification (manual on local test server).**
1. `/sfholo debug` spawns a text + icon + board hologram at the caller's
   feet. It is visible.
2. Walk past the configured range — it disappears.
3. Walk to the back side with `doubleSided=false` — it disappears.
4. Set `doubleSided=true` — it is visible from both sides.
5. Disconnect and reconnect — the hologram is re-shown without ghost entities.
6. Reload the plugin (`/sfreload`) — the hologram survives or is cleanly
   re-spawned, no orphaned entities.
7. On a 1.18.2 test server — the armor-stand backend is selected and the
   debug hologram renders (text only; icon line shows the item; board logs a
   "feature degraded" notice).

**Docs.** `docs/changes/<DATE>---holograms-phase-1-backends.md`.

**Ready when.** Build green, debug hologram renders on 1.21 Paper and on 1.18
Paper, both visibility rules behave, no entities leak on reload.

### Phase 3 — Text features (MiniMessage, placeholders, pages)

**Goal.** Hologram text supports legacy `&` codes always, MiniMessage when
Adventure is present (PlatformInfo#hasAdventureApi == true), per-player
placeholder expansion when PlaceholderAPI is installed, `%filler%`, and pages.

**Files to create.**

### Phase 4 — Custom packet layer + interactions

**Goal.** Click and hover dispatch on text lines, with cooldowns, command
execution as console or as player, and CText markup. The custom Netty layer
catches click packets; the renderer falls back to raycast if injection fails.

**Files to create.**
* `hologram/packet/PacketChannelInjector.java`. Steps:
    1. Resolve `CraftPlayer` class via `Bukkit.getServer().getClass().getPackage().getName()` (the `obc.v1_xx_Rx`-style path) — cached.
    2. Resolve `getHandle()` → ServerPlayer.
    3. Resolve the `connection` field (name varies per version: `playerConnection`, `b`, `c`, `connection`) — try a list; cache the winner per major MC version.
    4. Resolve `connection.connection` (the `Connection` / `NetworkManager`) — same approach.
    5. Resolve the `channel` field on `Connection` — same approach.
    6. Insert a `HologramPacketHandler` into the pipeline before `packet_handler` (the vanilla sink). Use a unique name like `sfcore-holograms`.
    7. On player quit, remove the handler.
* `hologram/packet/HologramPacketHandler.java` — `ChannelDuplexHandler`.
  `channelRead` checks the message's class name (no NMS imports). If it ends
  in `ServerboundInteractPacket` or legacy `PacketPlayInUseEntity`, hands it
  to `InteractDecoder`. Always calls `super.channelRead` so vanilla still
  receives the packet.
* `hologram/packet/InteractDecoder.java` — reflectively pulls the entity id
  field (`a` / `entityId`) and the action type (`type` / `b`). Resolves the
  hologram via `HologramRegistry`. If matched, posts `HologramClickEvent` on
  main thread via `SchedulerAdapter#runNow`.
* `hologram/packet/PacketAvailability.java` — owns the cached probe result.
  `HologramService` calls `PacketAvailability.probe()` once; subsequent calls
  return the cached value. On failure, the service logs once at INFO level
  and the renderer is told to use `RaycastTargeter`.
* `hologram/interact/InteractionDispatcher.java` — handles `HologramClickEvent`,
  resolves the CText action, cooldown-checks (`Map<UUID, Map<String, Long>>`),
  runs the command.
* `hologram/interact/RaycastTargeter.java` — ticks at 4 Hz (configurable),
  raycasts player look direction up to interaction-area depth, marks a hit.
* `hologram/interact/CTextParser.java` — pure string parsing, JUnit-testable.
  Outputs a list of `CTextSegment { text, command?, hoverText?, hoverColor? }`.
* `hologram/interact/ParticleFx.java` — spawn particles via Bukkit at the
  hologram location, scheduled via `SchedulerAdapter#runAtLocation`.

**Files to edit.**
* `Main.java` — register a `PlayerJoinEvent` / `PlayerQuitEvent` listener
  that calls into `PacketChannelInjector`. The listener lives in
  `hologram/listener/` and is registered alongside the existing join listener.
* `HologramService#load` — call `PacketAvailability.probe()` and log the
  result.
* `HologramSettings` — `interaction-enabled`, `interaction-area-width`,
  `interaction-area-height`, `cooldown-ms`, `click-particle`, `hover-particle`.
* `holograms.yml` — example entry showing CText markup.
* `HologramCommand` — new subcommand `interact <holo> enable|disable`.
* `language_en.yml` / `sprache_de.yml` — `hologram.interact.cooldown`,
  `hologram.interact.no_permission`, `hologram.interact.disabled`.

**Verification.**
1. On Paper 1.21, left-click a text line → its `<C>` command runs.
   Right-click → its right-click command runs. Cooldown is enforced.
2. Hover over a `<H>hover</H>` segment → color changes.
3. On a server where reflection probing fails (forced by editing the

4. No console errors on join/quit, even when the player disconnects before

5. The hologram packet handler does not interfere with chat, movement,

## 6. Custom Packet Layer — Detailed Design

This is the most reflection-heavy component, so it gets its own section.

### 6.1 Channel access (per-player)

```
Player p
  ↓ Bukkit
CraftPlayer
  ↓ getHandle()
ServerPlayer (NMS, name varies)
  ↓ field: "connection" / "playerConnection" / "c" / "b" (try in this order, cache winner)
ServerGamePacketListenerImpl (NMS)
  ↓ field: "connection" / "networkManager" / "a" / "h" (try in this order)
Connection (NMS, was NetworkManager pre-1.17)
  ↓ field: "channel" / "k" / "m" (try in this order)
io.netty.channel.Channel
```

Each step uses `Class#getDeclaredField` + `setAccessible(true)`. Field name
candidates are tried in order; the first non-null instance wins and the
chosen name is cached on the resolved class. If every candidate fails for any
step, `PacketAvailability` flips to `unavailable`, the error is logged once,
and the renderer uses raycast.

### 6.2 Pipeline insertion

```java
ChannelPipeline pipeline = channel.pipeline();
String anchor = "packet_handler"; // vanilla sink, present on all Paper/Spigot
String our = "sfcore-holograms";
if (pipeline.get(our) == null) {
    pipeline.addBefore(anchor, our, new HologramPacketHandler(/* deps */));
}
```

On `PlayerQuitEvent`, the handler is removed:
```java
ChannelPipeline pipeline = channel.pipeline();
if (pipeline.get("sfcore-holograms") != null) {
    pipeline.remove("sfcore-holograms");
}
```

Both operations run on the channel's event loop via
`channel.eventLoop().execute(...)` to avoid `ConcurrentModificationException`.

### 6.3 Packet recognition

We never `instanceof` an NMS class. Instead:

```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
    if (msg != null) {
        String name = msg.getClass().getSimpleName();
        if (name.equals("ServerboundInteractPacket") || name.equals("PacketPlayInUseEntity")) {
            InteractDecoder.decode(msg, this::onInteract);
        }
    }
    super.channelRead(ctx, msg);
}
```

`InteractDecoder` walks the packet's declared fields once per concrete class
(cached) to find an `int` field that holds the entity id and an `enum`-typed
field that holds the action (`INTERACT` / `ATTACK` / `INTERACT_AT`).

If the matched entity id is owned by a hologram in `HologramRegistry`, the
decoded event is dispatched onto the main thread via `SchedulerAdapter#runNow`.

### 6.4 Safety properties

* The handler **always** calls `super.channelRead`, so vanilla behavior is
  preserved 100%.
* The handler does **not** override `write`, so outbound packets are untouched.
* Exceptions from our handler are swallowed and logged at most once per
  player per minute; they never propagate into the Netty pipeline.
* On any reflection failure the layer is disabled at runtime — never at
  build time — and the renderer continues with raycast.

## 7. Display Entity Backend — Detailed Design

### 7.1 Loading
The backend class itself is fine to compile — all `org.bukkit.entity.TextDisplay`
references inside it are accessed only through `Class.forName` and
`Method.invoke`. The class file imports only standard Bukkit types
(`Entity`, `World`, `Location`, `ItemStack`) and `java.lang.reflect`.

The backend class is loaded only when `BackendSelector` returns it. This
means even if a server runs Java 8 on 1.12 (no module system at all), the
class loading itself can be deferred: the selector returns the armor-stand
backend without ever touching the display-entity class symbolically.

### 7.2 Entity spawning

```java
Class<?> textDisplayClass = Class.forName("org.bukkit.entity.TextDisplay");
Entity entity = world.spawn(location, (Class<? extends Entity>) textDisplayClass);
Method setText = textDisplayClass.getMethod("setText", Component.class);
Method setBillboard = textDisplayClass.getMethod("setBillboard", Class.forName("org.bukkit.entity.Display$Billboard"));
// ... etc
```

All `Method` handles are looked up once in a static initializer and cached.

### 7.3 Visibility

Paper exposes `Player#hideEntity(Plugin, Entity)` and `showEntity(Plugin,
Entity)` from 1.18 onward (the older single-arg variant is deprecated and
not used). The backend resolves both via reflection (`hasMethod` probe;
fall back to `Player#hidePlayer` semantics is **not** appropriate for
non-player entities — instead we drop to the armor-stand backend for that
server entirely).

### 7.4 Per-viewer text

When a `TextLine` template contains a per-player placeholder (detected by a
quick `template.contains("%")` check or by a placeholder probe), the backend
spawns one entity per viewer and hides each from the other viewers. For lines
without per-viewer content, a single shared entity is used.

The entity-per-viewer mode is bounded — if a hologram has > N viewers
(`per-viewer-soft-cap`, default 64), text is resolved with no player context
(server-wide placeholder values) and a single shared entity is used. Logged
once per hologram.

## 8. Armor-Stand Backend — Detailed Design

### 8.1 Constraints
* Custom name is the only "text". No background, no shadow, no alignment.
  Settings that don't translate are silently ignored.
* One stand per line.
* `setMarker(true)` → no hitbox → no client-side selection. Perfect for
  visual-only holograms. Click detection on this backend uses raycast only.
* Stands are summoned with metadata to make them resilient:
  `setRemoveWhenFarAway(false)`, `setCanPickupItems(false)`, etc.

### 8.2 Per-viewer text
Not supported on this backend. The text is resolved once at update time with
no player context. Detected per-viewer placeholders log a one-time warning
to the operator: "armor-stand backend in use; per-viewer text downgraded to
server-wide".

### 8.3 Icon lines
The stand wears the item on its head (standard Decent-style trick). For
small items, `setSmall(true)` is used to keep the head close to the line above.

## 9. Risk Register

| Risk | Phase | Mitigation |
|---|---|---|
| Reflective field names change between MC versions | 4, 7 | Name-candidate lists per step; first non-null wins; cached per resolved class. Fall back to raycast on full failure. |
| Display entity classloading breaks pre-1.19 servers | 1 | All `org.bukkit.entity.TextDisplay` references go through reflection; the backend class file imports only standard Bukkit types. |
| Folia region thread safety | 1, 5, 7 | All entity-affecting work goes through `SchedulerAdapter#runAtEntity` / `runAtLocation`. |
| Players see stale holograms after resource-pack reload | 1 | Dedicated `HologramResourcePackListener` re-spawns for that player on `PlayerResourcePackStatusEvent` reload events. |
| Hologram entities leak on `/reload` or plugin update | 1 | `HologramService#save` calls `backend.destroy(...)` for every registered hologram. Test the case. |
| Click detection breaks on alt-fork servers | 4 | Packet layer is opt-out automatically when reflection fails; raycast continues to work. |
| MiniMessage / Adventure missing on Spigot | 3 | `PlatformInfo#hasAdventureApi` gate; legacy `&` codes always available; MiniMessage syntax falls through as literal text with a one-time downgrade notice. |
| PlaceholderAPI missing | 3 | `PlaceholderBridge` is reflection-only; absence → identity function. |
| Per-viewer entity sprawl with many viewers | 7.x | `per-viewer-soft-cap`. Above the cap, shared text mode is used and an info log fires. |

## 10. Claude Code Session Playbook

### 10.1 Continuous-session kickoff prompt

Paste this as the first user message in a fresh session:

> Read `planning/brainstorming.txt` and `CLAUDE.md`. You are implementing the
> SF-Core holograms module from Phase 0 through Phase 7 **in this single
> session, continuously**. Follow §1 (constraints) absolutely. After each
> phase: run `./maven/bin/mvn -DskipTests package`, write the
> `docs/changes/<DATE>---*.md` entry per §5, update both language files,
> commit, then proceed to the next phase without waiting for approval. If a
> build fails, stop and surface the failure with the failing class names —
> do not continue. If a phase's verification list cannot be satisfied,
> stop with the same surfacing rule. Otherwise, run all the way to the
> end of Phase 7.

### 10.2 Build commands

Phase end: `./maven/bin/mvn -DskipTests package`

Quick check while iterating: `./maven/bin/mvn -DskipTests -o package`
(offline; faster after the first run).

### 10.3 Commit convention

One commit per phase. Suggested message format (mirroring existing log):

```
SF-Core Holograms: Phase N — <one-line summary>
```

The change-file's "Suggested Commit Message" section should hold this
verbatim.

### 10.4 Stop-and-surface rules

The agent stops the continuous loop and reports up exactly when:
1. The build fails after the agent's own edits in the current phase.
2. A verification item in the current phase's `Ready when` cannot be
   demonstrated (e.g. the server crashes on hologram spawn).
3. The agent discovers a constraint conflict — for example, the only
   sensible implementation path requires a new Maven dependency.
4. The agent must make a non-obvious decision the plan does not cover
   (rare — most decisions are spelled out above; if you reach one, surface
   it with options instead of guessing).

It does **not** stop to:
* Ask the operator to confirm style or naming inside the constraints.
* Ask for permission to write code inside the directories listed in §2.
* Ask whether to update the docs / language files (always yes).
* Ask whether to commit (yes, one per phase).

### 10.5 Rollback strategy

Each phase commits cleanly. To roll back, `git reset --hard <last good
phase commit>`. The `holograms.yml` file is created fresh in Phase 0, so
even a hard reset that leaves it behind on disk causes no harm — the
service ignores it when `enabled: false`.

## 11. Glossary (terms used in this plan)

* **Backend.** A concrete implementation of `HologramBackend`. Either
  `DisplayEntityBackend` or `ArmorStandBackend`.
* **Renderer.** `HologramRenderer` — backend-agnostic update entry point.
* **Viewer.** A `Player` that is allowed to see, and currently in range of, a
  hologram.
* **Per-viewer line.** A `TextLine` whose template depends on the viewer
  (placeholder, page, permission-filtered content).
* **CText.** The interaction markup: `<T>label</T><C>command</C><H>hover</H>`.
* **Dormant.** Service constructed and registered, but doing no work because
  `systems/holograms.yml → enabled: false`.
